请基于「抽取结果 JSON」「必填字段策略」和「用户原文」，分析当前工单的信息缺口。

输出 JSON 格式（不要 markdown 代码块）：
{
  "schemaMissing": ["environment", "timeRange"],
  "semanticGaps": ["未说明偶发还是必现"],
  "suggestedQuestions": ["问题是必现还是偶发？"],
  "blockingReason": "无法判断影响面",
  "readyForAnalysis": false,
  "confidence": 0.35
}

规则（重要）：
1. 【忠于抽取结果】schemaMissing 只能报「抽取结果 JSON 中对应字段为 null 或空」的字段。抽取结果里**已经有值**的字段，绝对不许重复报为缺失。
   - 例：抽取结果 affectedSystem="payment-service"，则 systemOrModule 不得出现在 schemaMissing 中。
   - 例：抽取结果 errorMessage="OOMKilled"，则 errorDetail 不得出现在 schemaMissing 中。
2. schemaMissing 的 key 仅使用：systemOrModule、apiOrFeature、environment、errorDetail、timeRange、impactScope
3. 【apiOrFeature 放宽】Pod/容器级别故障（OOMKilled、重启、CrashLoopBackOff 等 errorMessage 已体现）apiName 可缺失，apiOrFeature 不得作为阻塞项。
4. semanticGaps 描述模板外的语义缺口（批处理 Job 名、结算窗口、读写接口类型、复现频率等）。原文已明确的信息不要报为 gap。
5. suggestedQuestions 清晰、可回答，不要给出排障结论或根因猜测；只针对"真正缺失且影响分析"的项提问。
6. 【readyForAnalysis 核心判定——统一标准，宁宽勿严】
   判断原则：用户原文是否提供了足够支撑下一步分析（RAG 检索/工具取证/根因推理）的信息。
   满足以下任一条件即应返回 true（宁可放过，不要保守拦截）：
   - 用户原文已说明了**主体 + 现象/诉求**：知道是哪个系统/模块，且知道发生了什么（故障现象、或咨询的问题、或需求的内容）。
   - 抽取结果中 systemOrModule 非空，且 errorMessage 或 impactScope 或 apiName 中至少一个非空。
   - 即使时间/错误码等字段缺失，只要核心现象清晰（如"钱扣了订单没动"、"HikariPool 满了"、"问超时时间是多少"），就应返回 true。
   - 咨询/需求类（CONSULT/REQUIREMENT）主题明确即返回 true，不要追问环境/时间/影响范围（咨询类没有这些概念）。
   只有以下情况才返回 false（真正的信息不足）：
   - 严重缺失：连"是什么系统/什么问题"都说不清（如只说"接口报错了"、"不行了"、"挂了"），无法定位主体和问题。
   **关键：默认倾向 ready=true。** 后续有 RAG 取证和根因分析环节，能在分析中补充信息。口语化、错别字、顺序混乱、措辞不正式、缺少时间/错误码，都不构成 readyForAnalysis=false 的理由——只要知道"是什么系统+发生了什么"就算 ready。
7. 不要猜测未提供的信息。
